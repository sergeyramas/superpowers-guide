Зачем

Дефолтный агент — это junior без вкуса.
Superpowers даёт ему дисциплину.

Без Superpowers Claude (или Codex, или Gemini) на запрос «сделай X» сразу пишет код. Спека формируется в голове, тестов нет, после ревью агент покорно соглашается со всеми правками. Результат — итерации, регрессии, переделки.

// До Superpowers

  • ×«сделай фичу» → агент сразу кодит без вопросов
  • ×Спека держится в голове, не на бумаге
  • ×Тесты — либо нет, либо после кода
  • ×Агент говорит «готово» без проверки
  • ×На ревью соглашается со всем (performative agreement)
  • ×1 задача = 1 длинная последовательная сессия

// После Superpowers

  • ✓Сначала уточняет, что именно ты хочешь
  • ✓Чёткая спека, разбитая на читаемые куски
  • ✓Жёсткая red/green TDD — тест обязан упасть до имплементации
  • ✓verification-before-completion: evidence перед claim'ом
  • ✓Ревью обрабатывается технически, с верификацией каждой правки
  • ✓Параллельные сабагенты на независимые задачи
14 skills, 4 фазы

Skills срабатывают автоматически.
Никакого конфига.

Каждый skill знает, на какой фазе работы он включается. Пользователь ничего не выбирает — агент сам определяет нужный skill по контексту запроса.

// Discovery
using-superpowers Авто на старте сессии. Учит агента, как искать и применять остальные skills.
brainstorming Перед любой creative-работой — сначала уточняет, что ты хочешь.
writing-plans Превращает спеку в план для junior'а без контекста и вкуса.
// Execution
executing-plans Выполняет план в отдельной сессии с review-чекпоинтами.
subagent-driven-development Раздаёт задачи сабагентам в текущей сессии.
dispatching-parallel-agents Запускает 2+ независимые задачи параллельно вместо последовательно.
using-git-worktrees Изоляция feature-работы в worktree с safety-проверками.
// Quality
test-driven-development Жёсткая red/green: тест обязан упасть до начала кода.
systematic-debugging Системный сбор фактов вместо гадания при баге.
verification-before-completion Прогоняет команды и подтверждает вывод перед claim'ом «готово».
// Ship
requesting-code-review Запрашивает ревью перед merge.
receiving-code-review Технический rigor: верификация каждой правки, не performative.
finishing-a-development-branch Решает merge / PR / cleanup при готовых тестах.
writing-skills Создание / редактирование самих skills + проверка перед deploy.
Установка

Один из четырёх способов.
Самый простой — официальный marketplace.

Superpowers распространяется как plugin через несколько marketplace'ов. После установки агент сам подхватит skills — настраивать ничего не нужно.

# Самый стабильный путь — пакет проходит review Anthropic claude> /plugin install superpowers@claude-plugins-official # Готово. Skills подхватываются автоматически на старте.
# Свежие фичи раньше, чем в official marketplace claude> /plugin marketplace add obra/superpowers-marketplace claude> /plugin install superpowers@superpowers-marketplace
# В OpenAI Codex CLI codex> /plugins # Найти superpowers, установить
# Gemini CLI поддерживает через extension # см. gemini-extension.json в upstream-репо $ gemini extensions install obra/superpowers-marketplace
Сценарии

Как это выглядит в реальном диалоге

Три типичные ситуации — landing page с нуля, баг в проде, четыре фичи параллельно. Видны конкретные skills, которые срабатывают.

A Сделай мне новый landing page

ВЫ:    Сделай мне landing page для нашего продукта X.

CLAUDE: [триггер: brainstorming]
       Прежде чем писать код — давай разберёмся:
         1. Кто целевая аудитория?
         2. Какая главная боль?
         3. Какие 3 ключевых преимущества?
         4. Что должно произойти после клика на CTA?

ВЫ:    [отвечает]

CLAUDE: [триггер: writing-plans]
       План в 5 шагов:
         1. Hero: пейн → решение → primary CTA
         2. 3 преимущества с метриками
         3. Social proof (тестимониалы)
         4. FAQ
         5. Final CTA + footer
       Согласен начать с шага 1?

ВЫ:    Да.

CLAUDE: [триггер: TDD + executing-plans]
       Пишу падающий тест для CTA-ссылки. Запускаю → failed (ОК).
       Имплементирую CTA-компонент. Тест зелёный.
       [триггер: verification-before-completion]
       Прогоняю build, lint, тесты — всё ОК. Шаг 1 готов.

B У меня баг в проде

ВЫ:    Кнопка submit не работает в Safari.

CLAUDE: [триггер: systematic-debugging]
       Прежде чем гадать — соберу факты:
         1. В каких версиях Safari? Mobile / Desktop?
         2. Воспроизводится локально?
         3. Что в console?
         4. Какой последний коммит трогал submit?
       Дай мне эту информацию — потом сформулирую гипотезу.

C Реализуй 4 фичи параллельно

ВЫ:    Нужно: 1) добавить dark mode, 2) переделать footer,
       3) починить SEO-теги, 4) добавить sitemap.

CLAUDE: [триггер: dispatching-parallel-agents]
       Эти 4 задачи независимы (нет shared state).
       Запускаю 4 sub-agent'а параллельно, каждый в своём
       worktree. Через ~15 минут получишь 4 готовых PR.
FAQ

Частые вопросы

Это правда работает?

Да. Автор Jesse Vincent (obra) использует это в продакшне. На моих personal-проектах Superpowers + кастомные skills экономят 30-50% времени за счёт меньшего числа итераций и переделок.

Сжигает ли больше токенов?

Чуть-чуть на старт (системный промпт растёт на ~5-10K токенов) и на brainstorming-фазу. Но компенсируется за счёт меньшего числа неудачных попыток и переделок. Net-эффект — экономия на средних/крупных задачах.

Совместимо с моими собственными skills?

Полностью. Superpowers — отдельный плагин с префиксом superpowers: (например, superpowers:brainstorming). Не конфликтует с твоими skills в ~/.claude/skills/.

Можно отключить отдельные skills?

Да. Каждый skill — отдельный SKILL.md. Удалить или скопировать локально и редактировать триггеры.

А если я не хочу brainstorming на каждый запрос?

Skill срабатывает только на creative work (новые фичи / компоненты). Простые правки — не триггерятся. Если всё-таки слишком часто — отредактируй описание brainstorming/SKILL.md, чтобы триггер был уже.

Это open-source?

Полностью. MIT License. Можно форкать, модифицировать, использовать коммерчески. Этот гайд — тоже MIT.
